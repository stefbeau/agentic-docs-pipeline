ORCH
Orchestratorentry pointrouting
Receives the product change event, validates the payload, interprets the routing manifest, and dispatches downstream agents. Confidence-gates every stage transition.
confidence gatingpayload validationsequencing
Key behaviour: constructs the minimal execution plan from the routing manifest — not every agent fires on every run. A change requiring only a communications message dispatches Agent 04 only.

Confidence gating: low-confidence signals escalate to human review before any downstream agent fires. Early escalation prevents low-quality inputs from consuming pipeline compute.

Handoff isolation: agents do not communicate directly. All context flows through the Orchestrator. Every handoff package is constructed and logged here.
01
DET
Intake · Triagezero LLMdeterministic
Eight rule-based detectors classify the incoming change event and produce a typed routing manifest. Every routing decision is auditable and unit-testable.
8 detectorsfully auditablerouting manifest
Detectors: change_type · scope · audience · urgency · notify_trigger · content_trigger · review_trigger · escalation_trigger

Why no LLM: routing is the most consequential decision in the pipeline. A routing error propagates through every downstream stage. Deterministic detectors make routing errors rare, visible, and debuggable.

Uncertain outputs: uncertain is a valid output for notify and content triggers — not an error. Ambiguous signals route to confidence gating rather than forcing a wrong binary answer.
02
LLM
Impact Analysisvector searchseverity scoring
Semantic search across the documentation corpus. Identifies affected pages, scores by severity, and produces a ranked impact manifest with per-page change rationales.
corpus searchseverity: critical · high · medium · low
Search strategy: scope-parameterised vector search. Single-product changes search within the affected content area; platform-wide changes search the full corpus.

Conservative threshold: the agent errs toward inclusion. A false positive (incorrect inclusion) surfaces at the approval gate and is dismissed. A false negative (missed page) is a silent documentation gap.

Change rationale: each affected page receives a one-sentence rationale explaining why it is flagged — surfaced to the writer at the approval gate for fast triage.
03
LLM
Content Prepdiff outputMCP retrieval
Drafts documentation updates as structured DIFF packages — current content, proposed content, rationale, confidence score. Live page retrieval via MCP tool call.
live page fetchstructured DIFFdependency flagging
Minimal change principle: the agent makes the smallest change that accurately reflects the product update. No restructuring, no style improvements beyond scope.

Live retrieval: page content is fetched at runtime via MCP — not from cache. Drafts are always based on the current corpus state.

DIFF format: structured JSON with current_content, proposed_content, change_rationale, and confidence. Writers see exactly what changes — not a full page rewrite.
04
LLM
Communicationsvoice constrained2 sub-agents
Drafts client-facing communications messages. Pre-script handles template selection and variable extraction; two sub-agents handle drafting and voice normalisation.
pre-scriptvoice normalisationpost-script envelope
Pattern C applied: deterministic pre-script extracts and structures inputs before the LLM fires. Deterministic post-script constructs the output envelope. LLM budget reserved for drafting and normalisation only.

Two sub-agents: draft generation and voice normalisation are separate cognitive tasks. A single agent asked to do both tends to over-normalise during drafting or under-normalise during review.

No send capability: the agent drafts only. Email dispatch is a post-approval action outside the pipeline.
✋ Human Approval Gate
Pipeline stops here. Writer reviews the impact manifest, content DIFF package, and communications draft. Every component requires an explicit approve / edit / reject decision. Nothing proceeds without it. This gate is architectural — not a policy toggle.
05
LLM
Quality Review77 rules3 sub-agents
Evaluates approved content against a structured rule corpus — 39 prose rules, 19 style rules, 19 terminology rules. Three parallel sub-agents, one per category.
parallel executionrule-referenced findingstraceable to rule IDs
Parallel sub-agents: prose, style, and terminology sub-agents run concurrently. No inter-agent dependencies. Total latency equals one sub-agent call, not three sequential calls.

Rule-referenced findings: every finding is tagged with a rule ID, severity level, location, and remediation suggestion. Findings are machine-readable — not a paragraph of feedback.

Why post-gate: running quality review before approval would evaluate content the writer may edit. Post-gate review evaluates the content the writer actually approved.
06
GOV
Governanceaudit loghealth metrics
Assembles the complete run log, detects anomalies against thresholds, publishes health metrics, and writes the append-only corpus change record.
run loganomaly detectioncorpus change record
Append-only record: the corpus change record cannot be edited. Every published update is traceable to the product change event that triggered it, the writer who approved it, and the timestamp.

Anomaly thresholds: token consumption, pipeline duration, confidence scores, approval edit rate, quality findings severity — all evaluated against baselines. Flagged runs surface in the health dashboard.

Maintenance signal: high approval edit rates on a specific agent's output are the clearest signal that a prompt needs tuning. Governance makes that signal visible.

Design Principles
Pattern C — Deterministic-first
Determinism where cheap, LLM where judgment required. Triage layer uses zero LLM.
Human-in-the-loop by design
Approval gate is architectural — not a policy toggle, not bypassable.
Bounded agent scope
Each agent has one job. No agent knows the full pipeline state.
Audit-first
Every run produces a complete, traceable log. Nothing is a black box.
